# Issue Creator

`issues/<slug>/issue.md` を生成し、調査・タスク・メモ・設計を軽量に記録する。

## 実行順序（必須）

**依頼を受けたら、調査・回答・実装などのいかなる作業にも着手する前に、必ず先に `issues/<slug>/` と `issue.md` を作成する。** ファイル作成を後回しにしない。

## いつ使うか

- 仕様駆動（Kiro）までしない一時的な調査・タスク・メモ・設計メモ
- 発見やメモをフォルダ単位で残したいとき

**Kiro との使い分け**: Kiro = 要件が確定し、実装・設計・レビューフローが必要。Issue Creator = 軽い調査・メモ・タスク・設計のメモ。

## ワークフロー

### ステップ1: ヒアリング（必須）

ユーザーの依頼から以下を確認する。

1. **目的の種類**（いずれか／複数可）
   - `タスク` - 実施予定の簡易タスク
   - `メモ` - リンク・発見・参考情報の記録
   - `調査` - 既存機能・バグ・パフォーマンス等の調査
   - `設計` - 設計メモ・方針・検討事項

2. **実行対象**
   - どのリポジトリ／プロジェクトの `issues/` に置くか。ヒアリングで確認する。

3. **タイトル**
   - 日本語で簡潔に（10〜20字程度）。例: "DB クエリ調査", "ログイン画面の設計メモ"

### ステップ2: slug を決める

- タイトルから英訳 or ローマ字化し、ハイフン区切り・小文字・英数字のみ（`[a-z0-9-]+`）
- 例: "DB クエリ調査" → `investigate-db-query`、"ログイン設計メモ" → `design-login-screen`
- 既に `issues/<slug>/` がある場合は別 slug を提案するかユーザーに確認する

### ステップ3: まずファイルを生成してから内容を埋める

- **他作業の前に、必ず先に** `issues/<slug>/issue.md` を作成する
- `assets/issue.md` をベースに、タイトル・作成日（YYYY-MM-DD）・Category（タスク／メモ／調査／設計）・実行対象（ヒアリング結果）・概要・必要なら各セクションを記入。不要なセクションは削除してよい

## テンプレート

`assets/issue.md` を読み、その構造で `issue.md` を生成する。目的に応じてセクションを増減してよい（調査→調査結果、タスク→タスク、設計→設計メモ・方針など）。

## 注意事項

- セクションは不要なら削除してよい。独自セクションの追加は自由
- Status は初期値 `open`、日付は作成日
- 複数目的の場合は該当するセクションをすべて含めてよい
